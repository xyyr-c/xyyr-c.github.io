0x03 80 字节墙分析

每次 read 写入的 80 字节覆盖范围：

rbp+0x10 是第一个 ROP 延续，也就是第 80 字节——刚好写入一个字节都不多！

这意味着单次 vuln 的 leave;ret 之后，我们永远只能控制：

  1. saved_rbp（被 leave pop 进 rbp）
  2. ret_addr（被 ret pop 进 rip）

ROP 链无法在单次溢出内完成。

0x04 解题思路：双 BSS Pivot + libc 泄露

由于题目没有提供 libc 地址泄露，且远程环境下无法通过 /proc/ 获取基址，我们需要先利用双 BSS Pivot 泄露 libc 基址，再第二轮利用获取 shell。

核心技巧：用两次 BSS 写入在不同偏移，让第一次写入的数据在第二次写入后仍然保留。

泄露思路： 通过 puts(puts@got) 打印 GOT 表中的 puts 实际地址。puts@got 存放的是 libc 中 puts 函数的运行时地址（6 字节，48-bit），将其作为字符串输出即可泄露。

  • BSS1 = 0x404800 — 用于第一轮泄露链
  • BSS2 = 0x404A00 — 用于第二轮 system 链（与 BSS1 不重叠）

0x07 总结

难点 解决方案
80 字节文件写入限制 双 BSS pivot，利用写入偏移不重叠保留数据
栈迁移目标地址不可用 先通过 VULN_MID 写入 BSS 预填，再 leave;ret 迁移
system 栈对齐 RET gadget 对齐 rsp 到 16n+8
pop rdi 需要两个参数 BSS 预填 POP_RDI + binsh_addr
无 libc 基地址泄露 第一轮用 puts(puts@got) 打印 libc 地址，第二轮用泄露值计算 system

核心创新： 第一次 BSS 写入在 BSS+0x60 放 POP_RDI 和后续参数，第二次写入从 BSS+0x10 开始，leave 从 BSS+0x58 取 RET，BSS+0x60 正好是第二次写入的第 80 字节——完美避开了覆盖。

0x08 第二种写法：LEAVE_RET 精简方案

在分析过程中探索了另一种更精简的写法：利用 LEAVE_RET（0x40127b）直接迁移到预写的 system 链，省去第二轮 double BSS pivot 的部分步骤。

思路： 泄露链末尾的 POP_RBP 不再跳到 VULN_MID，而是跳到 LEAVE_RET，直接迁移到 BSS1+0x10——该位置在第二轮写入时已被覆盖，可以预埋 system 链。

约束： system 链需要 libc 地址（system、/bin/sh），而泄露发生前不知道基址。此方案仅适用于已知 libc 基址的场景（如本地调试时通过 /proc/pid/maps 获取，或 ASLR 关闭时）。

0x09 两种方案对比

特性 原始双 BSS pivot LEAVE_RET 精简版
远程可用 ✅ 完全独立泄露 ❌ 需先知 libc 基址

代码行数 较长 更短

原始方案 适用于远程场景，完全通过 puts(puts@got) 泄露获取 libc 基址后二次写入。

LEAVE_RET 方案 利用 leave; ret 的栈迁移能力直接跳转到预埋数据，结构更紧凑，但需要预先知道 libc 地址。适合本地调试或 ASLR 关闭的场景。
